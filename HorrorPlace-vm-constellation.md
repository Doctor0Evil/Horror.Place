## Declarative wiring

Cross-repository linkage is expressed declaratively through schemas, registries, and DSLs instead of imperative glue code. Every object in the constellation—events, regions, personas, styles—is defined by a JSON schema in the public core, such as `schemas/invariants_v1.json`, `schemas/entertainment_metrics_v1.json`, and `schemas/stylecontract_v1.json`. [ppl-ai-file-upload.s3.amazonaws](https://ppl-ai-file-upload.s3.amazonaws.com/web/direct-files/collection_cdb90fc3-8a6a-46e2-89a6-187b2f85f988/842c2d27-18c3-4246-8040-11c76bb58157/a-new-addition-to-the-rivers-o-kBKpoeZeQ9mj8PyJlgtxSg.md)

Registries like `registry/events.json`, `registry/regions.json`, `registry/styles.json`, and `registry/personas.json` map stable IDs to contract locations and metadata rather than raw file paths. Runtime systems and AI agents resolve objects by ID through these registries, which allows asset locations to change without breaking consumers. ## Narrow APIs and engine role

Engine-facing access to the constellation is mediated through narrow, versioned APIs instead of direct repo coupling. The public core exposes stable Lua-facing contracts like `H.CIC(region_id)`, `H.metrics(event_id)`, and `H.resolve_event(event_id)` that resolve through registries and never leak raw underground content.
